# darktitan import

Bring external resources into DarkTitan. Two subcommands: `import tasks` for Markdown task files, and `import flow` for flow YAML definitions.

## darktitan import tasks

### Synopsis

```bash
darktitan import tasks <path_to_tasks_file> [project_name]
```

### Description

Imports tickets from a Markdown tasks file into DarkTitan's Thinking Department. The file is typically generated by `darktitan new tasks`, but any Markdown file following the expected format works.

The command previews what will be imported (task count and phases), asks for confirmation, then creates the tickets in bulk. All tickets are created with status `THINKING`.

**The DarkTitan server must be running** (`darktitan run` or `darktitan ui`) before you can import.

### Arguments

| Argument | Required | Description |
|---|---|---|
| `<path_to_tasks_file>` | Yes | Path to the Markdown file containing tasks |
| `[project_name]` | No | Project to assign tickets to. Defaults to the active project set in `settings.json`. |

### Example session

```bash
$ darktitan import tasks tasks.md

Found 18 tasks across 3 phases:
  Phase 1: Foundation                          5 tasks
  Phase 2: Core Features                       9 tasks
  Phase 3: Polish                              4 tasks

Import all 18 tasks to THINKING in project "my-app"? [y/N] y
✓ 18 tickets created in Thinking Department
```

### Task file format

The importer reads a structured Markdown file. Each `- [ ]` line becomes one ticket. `## ` headings set the group name. Metadata lines (`Priority`, `Description`, `Label`) are optional but recommended.

A minimal example (`docs-content/examples/tasks.md`):

```markdown
## Phase 1: Setup

- [ ] Initialize project structure
  - Priority: high
  - Description: Create the initial directory layout, add a README, and set up the main entry point.
  - Label: implement

- [ ] Add unit tests for core logic
  - Priority: medium
  - Description: Write tests covering the happy path and main error cases for the core module.
  - Label: implement
```

| Field | Values | Description |
|---|---|---|
| `Priority` | `high`, `medium`, `low` | Ticket priority |
| `Description` | Any text | One-sentence description of the task |
| `Label` | Flow name | If set, the router uses this as a fast-path to assign the ticket directly to that flow |

### Notes

- The server must be running on the configured port (default `7700`). If it is not, you will see: `DarkTitan server not running. Start it with: darktitan run`.
- If no `[project_name]` argument is given, the active project from `settings.json` is used. If no active project is set, tickets are created without a project association.
- You can import the same file multiple times — duplicate detection is not automatic, so review before confirming.

## darktitan import flow

### Synopsis

```bash
darktitan import flow <flow-yaml> <project-name>
```

### Description

Copies a flow YAML file into `~/.darktitan/flows/` (so the worker can find it) and links it to an existing project in SQLite. If the file is already inside the flows directory, the copy step is skipped and only the project link is recorded.

The project must already exist (created via `darktitan new project` or the API).

### Arguments

| Argument | Required | Description |
|---|---|---|
| `<flow-yaml>` | Yes | Path to a flow YAML file on disk (any location) |
| `<project-name>` | Yes | Name of an existing project to link the flow to |

### Example session

```bash
$ darktitan import flow ./pipelines/implement.yaml my-app
✓ Copied implement.yaml to /Users/you/.darktitan/flows
✓ Linked implement.yaml to project my-app
```

### Typical workflow

```bash
# 1. Author a flow YAML in your editor (or copy from another project)
$EDITOR implement.yaml

# 2. Validate it
darktitan validate implement.yaml

# 3. Import it into DarkTitan and link to a project
darktitan import flow implement.yaml my-app

# 4. The router will pick up tickets labelled "implement" assigned to my-app
```

### Notes

- The flow filename (without path) becomes the canonical name used by the router and `assigned_flow_name` field.
- Re-importing a flow YAML with the same filename overwrites the existing copy in `~/.darktitan/flows/` and re-records the project link (idempotent).
- Project linkage lives in the SQLite `project_flows` table; no YAML manifest is updated.
